## 2. 它是什么

`CodexPotter` 不是普通聊天工具，而是一个多轮任务执行器。

可以把它理解为：

- 你给它一个明确任务
- 它调用本机 `codex`
- 每一轮都用新的上下文重新推进任务
- 自动根据文件产物、计划文档、已有修改继续收敛结果

它更适合：

- 明确目标的开发任务
- 自动修复、重构、补测试
- 先出方案，再按方案实现

它不适合：

- 前端样式需要你频繁人工点评的任务
- 来回追问式聊天
- 纯问答、纯头脑风暴

## 3. 最快开始

在项目目录里打开终端，执行：

```powershell
cd C:\Users\GaoqQang\Desktop\mypc\CodexPotter
.\codex-potter-local.cmd --yolo
```

启动后，直接输入一个任务，不要按聊天方式提问。

推荐这样写：

```text
分析当前仓库，找出首页接口响应慢的主要原因并直接修复。
完成后运行相关测试，并把关键结论写入 docs/perf_fix.md。
```

不推荐这样写：

```text
你觉得这里慢吗？
```

原因很简单：`CodexPotter` 擅长执行任务，不擅长和你来回聊天。

## 4. 最推荐的使用方式

### 方式一：直接执行一个完整任务

```powershell
.\codex-potter-local.cmd --yolo
```

然后输入：

```text
清理当前仓库中的重复逻辑，删除无用文件，修复明显错误，并确保测试通过。
```

适合一次性推进完整工作。

### 方式二：先规划，再实现

第一步，先生成方案：

```text
分析当前仓库，设计一个更简单的商品筛选方案。
把方案写入 docs/filter_redesign.md，不要实现。
```

第二步，再实现：

```text
根据 docs/filter_redesign.md 实现改造。
补齐受影响的测试，并确保通过。
```

这是最稳妥的用法，因为方案文件会成为后续多轮执行的稳定依据。

### 方式三：把关键约束写入文件

如果任务复杂，不要只靠一句临时输入。最好把要求写入仓库文件，例如：

- `docs/xxx.md`
- `DESIGN.md`
- `MAIN.md`

然后让它按文件执行：

```text
根据 docs/filter_redesign.md 实现，不要偏离文档约束。
```

这样能明显减少跑偏。

## 5. 常用命令

### 5.1 交互式运行

```powershell
.\codex-potter-local.cmd --yolo
```

常见变体：

```powershell
.\codex-potter-local.cmd --rounds 5 --yolo
.\codex-potter-local.cmd --strict-rounds 100 --yolo
.\codex-potter-local.cmd --search --yolo
.\codex-potter-local.cmd -m gpt-5.4 --yolo
```

说明：

- `--rounds 5`：最多执行 5 轮
- `--strict-rounds 100`：严格跑满 100 轮，设置后会覆盖 `--rounds`
- `--search`：允许联网搜索
- `-m gpt-5.4`：指定模型
- `--yolo`：尽量跳过审批和沙箱阻拦，适合本机全自动执行
- 不写 `--strict-rounds`：行为和原版一样

### 5.1.1 严格跑满指定轮数

如果你要“不是最多 N 轮，而是必须完整跑满 N 轮”，用：

```powershell
.\codex-potter-local.cmd --strict-rounds 100 --yolo
```

这和 `--rounds 100` 不一样：

- `--rounds 100`：最多 100 轮，任务提前完成可能提前停
- `--strict-rounds 100`：强制按 100 轮预算跑完，前面轮次即使出现完成标记也不会提前结束

如果同时写了：

```powershell
.\codex-potter-local.cmd --rounds 20 --strict-rounds 100 --yolo
```

实际以 `--strict-rounds 100` 为准。

### 5.1.2 启动后的实际执行方式

执行下面这条命令以后：

```powershell
.\codex-potter-local.cmd --strict-rounds 100 --yolo
```

它仍然会先让你输入一次任务，然后才开始自动执行。

例如你输入：

```text
分析当前仓库的订单流程，修复明显错误，补齐关键测试，并把结论写入 docs/order_flow_fix.md。
```

后续这 100 轮，不会让你再手动输入 100 次任务，而是围绕这一次输入的目标持续推进。

更准确地说，它是围绕同一个项目记录文件继续执行，而不是靠聊天上下文硬记住你前面说过的话。所以重要约束最好写进文件，或者在第一次任务里写完整。

你可以把它理解成：

- 启动一次，输入一次任务
- 自动连续跑多轮
- 每轮都继续推进同一个项目目标
- 跑完后退出

另外要注意：

- `--strict-rounds 100` 的意思是忽略“提前完成”标记，直到第 100 轮才允许按这个规则结束
- 它不等于“无论发生什么都一定能跑完 100 轮”
- 如果中途你手动中断、`codex` 报错、进程异常退出，仍然会停

### 5.1.3 中断与异常日志怎么看

现在每个项目目录里除了原来的 `potter-rollout.jsonl`，还会额外写一个：

```text
.codexpotter/projects/YYYY/MM/DD/N/potter-runtime.jsonl
```

这个文件专门记录“启动/轮次开始/异常原因”，用于排查为什么没有继续跑下去。

常见记录类型：

- `session_started`：这次启动或 resume 已经开始
- `round_started`：某一轮真的开始执行了
- `diagnostic`：出现了中断或异常

常见原因字段 `reason`：

- `manual_interrupt`：手动中断
- `event_stream_closed`：事件流异常关闭
- `backend_spawn_failed`：后端 `codex app-server` 没拉起来
- `runtime_error`：其它运行期错误

排查时建议这样看：

1. 先看 `potter-runtime.jsonl` 最后几行，确认停在第几轮、原因是什么。
2. 再看同目录下的 `potter-rollout.jsonl`，确认这一轮有没有写到 `round_configured`、`round_finished`。

如果你遇到“明明设了 `--strict-rounds 100`，却中途停了”，优先看：

- 有没有最后一条 `diagnostic`
- `reason` 是不是 `manual_interrupt`、`event_stream_closed` 或 `backend_spawn_failed`
- 最后一条 `round_started` 是第几轮

补充说明：

- `potter-rollout.jsonl` 仍然是项目进度主记录
- `potter-runtime.jsonl` 是额外的诊断侧日志，不影响 resume 语义

### 5.2 继续之前的任务

```powershell
.\codex-potter-local.cmd resume
```

它会读取以前的任务记录并继续推进。

也可以直接指定某个任务：

```powershell
.\codex-potter-local.cmd resume .codexpotter/projects/2026/04/12/1
```

任务记录通常在：

```text
.codexpotter/projects/YYYY/MM/DD/N/MAIN.md
```

### 5.3 非交互执行

适合脚本或一次性任务：

```powershell
.\codex-potter-local.cmd exec "修复当前 failing tests"
```

或从标准输入读取：

```powershell
"检查仓库结构并生成简化建议" | .\codex-potter-local.cmd exec
```

如果要机器可读输出：

```powershell
.\codex-potter-local.cmd exec --json "修复当前 failing tests"
```

## 6. 如何写任务提示词

好的任务提示词要满足四件事：

- 目标明确
- 范围明确
- 输出明确
- 验证明确

推荐模板：

```text
目标：修复当前仓库中的登录失败问题。
范围：只改动登录相关代码和必要测试，不要重构无关模块。
输出：修复问题，补充测试，并将原因写入 docs/login_fix.md。
验证：运行相关测试并确保通过。
```

更简化一点也可以：

```text
修复登录失败问题，只改登录相关代码。
补测试并运行通过。
把根因和修复方式写入 docs/login_fix.md。
```

## 7. 非常重要的行为特性

### 7.1 每次后续输入都是新任务

这点最容易踩坑。

`CodexPotter` 和聊天式 `codex` 不一样。你后面再输入一句话时，它不会默认继承刚才完整对话上下文，而是更偏向“开启一个新的任务回合”。

但这里有一个容易混淆的点：

- 同一次启动后的多轮执行，仍然是在推进同一个项目
- 退出以后你再输入新的提示词，才会变成新的任务

所以：

- 不要依赖“上一句我已经说过”
- 重要要求要写进文件
- 方案、约束、验收标准最好落盘

### 7.2 它更像自动推进器，不像讨论助手

如果你要的是：

- 一边看 UI 一边让它改
- 改一点你点评一点
- 不断闲聊式探索

那更适合直接用 `codex`，不是 `codex-potter`。

## 9. 常见参数说明

```powershell
codex-potter --help
codex-potter resume --help
codex-potter exec --help
```

你最常会用到这些参数：

- `--yolo`：全自动执行，少阻塞
- `--rounds N`：限制最大轮数
- `--strict-rounds N`：严格跑满 N 轮，并覆盖 `--rounds`
- `--search`：允许搜索网络资料
- `-m MODEL`：指定模型
- `--sandbox workspace-write`：以工作区可写方式运行
- `--codex-bin <PATH>`：指定 `codex` 可执行文件路径

## 10. 建议的工作习惯

为了让结果更稳定，建议固定用下面这套节奏：

1. 先定义任务目标
2. 把关键要求写入文档文件
3. 用 `codex-potter --yolo` 执行
4. 如需继续，使用 `codex-potter resume`
5. 重要结果继续沉淀到文件，而不是只留在终端输出里

补充：

- 如果你准备使用 `--strict-rounds xxx` 长轮次推进，建议直接套用同目录模板：
  - `CodexPotter使用说明/MAIN模板-严格跑满xxx轮.md`
- 这个模板已经把“同一项目持续推进”“发现新问题后如何落盘到下一轮继续处理”“如何避免 100 轮跑偏”写进去了
- 最稳的方式不是重复补提示词，而是持续维护 `MAIN.md`
