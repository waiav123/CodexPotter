# Overall Goal

围绕 `plati.market ↔ 闲鱼` 跨平台虚拟商品套利管理系统持续推进，不开启独立支线任务。

本次执行目标：

- 打通并收敛主链路：`采集 -> 入库 -> 翻译 -> 匹配 -> 价差计算 -> 工作台展示/管理`
- 优先补齐阻塞主链路的真实缺口，不做无关扩散
- 每轮都必须基于当前代码、文档、数据库结构和本文件状态继续推进

建议启动命令：

```powershell
.\codex-potter-local.cmd --strict-rounds xxx --yolo
```

如果不设置 `--strict-rounds`，则按原版“最多 N 轮 / 可提前结束”的语义执行。

## Hard Constraints

- 所有轮次都围绕本文件定义的同一个项目目标推进，不要把中途发现的问题拆成新的独立任务。
- 所有关键约束、阶段结论、阻塞项、待办项都必须落盘到本文件或仓库文档，不要只保留在终端输出里。
- 所有运行时配置统一以仓库根 `.env` 为准，不新增其它配置真值源。
- 不为旧行为、旧接口、旧流程保留兼容层，默认直接迁移到新实现。
- 优先保持主链路简单、清楚、低噪音，不增加无必要页面层级、状态分叉、按钮和补丁式逻辑。
- 改动优先服务于真实业务闭环：先保证主链路能跑通，再做增强，不做脱离主链路的装饰性开发。
- 若发现新问题：
  - 阻塞当前主链路的，写入 `In Progress` 并优先解决
  - 不阻塞但必须后续处理的，写入 `Todo`
  - 已完成的，写入 `Done`，并附验证结果

## Round Discipline

- 本文件是后续所有轮次共享的项目状态入口。
- 每一轮开始时，优先阅读并遵循本文件的 `In Progress`、`Todo`、`Done`。
- 每一轮结束前，必须至少完成以下动作之一：
  - 更新代码
  - 更新文档
  - 更新本文件状态
  - 补充测试或验证记录
- 即使某个局部问题看起来已经完成，只要仍在严格轮次预算内，也要继续做以下事项之一：
  - 主链路缺口排查
  - 回归验证
  - 文档收口
  - 死代码/重复逻辑清理
  - 异常恢复与可观测性补强
- 如果发现“这一轮想做的事”没有落到文件或代码里，视为下一轮不可靠，不要依赖临时记忆。

## Current Focus

当前阶段优先级：

1. 保证 `plati` 商品采集与入库链路完整、字段准确、可持续执行
2. 保证翻译链路对未翻译数据可自动处理，且不覆盖原始字段
3. 保证 `plati ↔ 闲鱼` 商品匹配与价差计算链路可稳定产出可操作结果
4. 保证工作台支持快速筛选、查看、管理套利机会
5. 保证关键异常可重试、可落盘、可排查

## Acceptance

- 采集主链路可稳定运行，关键字段完整且无明显污染
- 翻译链路可识别未翻译内容并完成状态流转，原始数据与翻译结果分字段隔离
- 匹配与套利筛选链路可以产出可验证结果，不是只停留在方案层
- 关键接口、服务或页面改动必须补齐必要测试，或补充明确的人工验证记录
- 关键结论、设计决策、已知限制必须同步写入仓库文档
- 严格轮次内如果发现新的结构性缺口，不能只口头提及，必须写入 `Todo` 或 `In Progress`

## Preferred Working Order

每轮默认按这个顺序决策：

1. 先处理 `In Progress` 中阻塞主链路的问题
2. 再处理 `Todo` 中最影响主链路收敛的一项
3. 完成后补测试、补验证、补文档
4. 如果主链路暂无新缺口，转入清理重复逻辑、补强异常处理和完善可观测性

## In Progress

- 当前正在处理的主问题写这里。
- 写法建议：
  - 问题是什么
  - 为什么阻塞主链路
  - 本轮准备怎么收敛

示例：

- 补齐 `plati` 采集结果到数据库的字段映射，避免多级分类和子商品价格丢失，当前优先核实入库口径与现有 schema 是否一致。

## Todo

- 这里放“已经确认值得做，但不是本轮最优先”的事项。
- 严格轮次模式下，新发现但暂不处理的问题，一律写到这里，供下一轮继续。

示例：

- 评估 `plati` 全量采集的增量同步策略，降低重复抓取成本。
- 收紧翻译任务扫描条件，避免重复扫描已完成数据。
- 为套利工作台补充高价值筛选维度，减少人工筛查成本。
- 统一运行异常日志口径，确保 `manual_interrupt`、`event_stream_closed`、`backend_spawn_failed`、`429` 等原因都可直接落盘排查。

## Done

- 这里记录已经完成并验证过的事项。
- 每条建议包含：
  - 做了什么
  - 为什么这样做
  - 验证方式
  - 涉及文件

示例：

- 为批量导入链路补齐原子提交语义，避免半成功半失败导致数据状态不一致。
  - 决策：仅收紧事务边界，不新增兼容逻辑。
  - 验证：运行相关测试并通过。
  - 文件：`web_admin/service.py`、`tests/test_arbitrage_service.py`
